# AI Workflow Rules — BlueCollar Job Portal

These are rules for an AI coding agent working on this codebase. Follow them without exception. Do not reinterpret them as suggestions.

---

## 1. Overall Approach

**Work spec-first, code second.**
Before writing any implementation code for a unit of work, state what you are about to build, which files you will create or modify, and what the expected behavior will be. Do not begin coding until that plan is confirmed.

**Follow the architecture.md and project-overview.md as ground truth.**
If your implementation conflicts with a decision documented in `architecture.md` or `project-overview.md`, stop and flag the conflict. Do not resolve it by silently deviating from the documented decision.

**Treat the Prisma schema as the source of truth for data shapes.**
Do not infer field names, types, or relations from usage in controllers or services. Read `prisma/schema.prisma` before writing any code that touches the database.

**Never generate speculative code.**
Do not add fields, endpoints, components, or utilities that are not required by the current unit of work. If you think something will be needed later, say so — do not build it unasked.

---

## 2. Scoping Rules

**Build one unit at a time.**
A unit is one of the following: a single API route + controller + service, a single React page, a single middleware function, or a single Prisma migration. Do not work on two units simultaneously.

**Do not modify files outside the current unit.**
If implementing `POST /api/jobs` requires a change in `authMiddleware.js`, stop and flag it before touching that file. Only modify files that are directly required for the current unit to function.

**Do not refactor while implementing.**
If you notice a structural problem in existing code while implementing a new unit, note it separately. Do not refactor and implement in the same step. Mixing the two makes diffs unreadable and bugs harder to isolate.

**Do not add dependencies without stating why.**
Before installing any new npm package, name the package, state what it does, and confirm it is not already handled by an existing dependency in the project. Do not install packages as a convenience if the same result can be achieved with what is already installed.

**Do not create placeholder files or TODO stubs.**
Every file you create must be functional at the time of creation. A stub that exports an empty function and promises to be filled in later is a broken unit. Do not commit it.

---

## 3. When to Split Work into Smaller Steps

**Split when a unit touches more than three files.**
If completing a unit requires creating or modifying more than three files, decompose it into sub-steps and list them before starting. Complete and verify each sub-step before moving to the next.

**Split when a unit spans both the frontend and backend.**
Implement and verify the backend route first. Confirm the API response shape is correct. Then implement the frontend that consumes it. Never build both simultaneously.

**Split when a database schema change is involved.**
A Prisma schema change is always its own step. Run `prisma migrate dev`, confirm the migration was applied, and confirm existing data (in development) is intact before writing any code that depends on the new schema.

**Split when an invariant from `architecture.md` is at risk.**
If completing a unit in one step would require temporarily violating an invariant (e.g. putting a Prisma query in a controller to test something quickly), split the work so the invariant is never violated, even temporarily.

---

## 4. Handling Missing or Ambiguous Requirements

**Do not assume. Ask before building.**
If a requirement is missing — for example, what happens when a worker applies to a job they already applied to — stop and ask. State the question specifically and wait for a confirmed answer before writing code.

**Do not infer behavior from similar features.**
The fact that one endpoint returns a 404 when a resource is not found does not mean another endpoint should do the same. Each unit's error behavior must be specified or confirmed.

**When two interpretations are equally valid, present both and wait.**
Do not pick one silently. Write out the two options, state the tradeoff between them, and ask which to implement.

**Do not use placeholders for unresolved requirements.**
If a field's behavior is unclear, do not use a hardcoded default or a null value and move on. Treat the ambiguity as a blocker and surface it before continuing.

---

## 5. Protected Files

**Do not modify these files without an explicit instruction to do so:**

| File / Pattern | Reason |
|---|---|
| `prisma/migrations/**` | Auto-generated by Prisma. Editing by hand corrupts the migration history. Run `prisma migrate dev` to generate new migrations. |
| `prisma/schema.prisma` | Schema changes require an explicit instruction. Do not add or remove fields as a side effect of implementing another feature. |
| `tailwind.config.js` | Design tokens. Changes here affect the entire UI. Only modify when explicitly changing the design system. |
| `vite.config.js` | Build configuration. Incorrect changes break the build silently. Only modify with explicit instruction. |
| `package.json` / `package-lock.json` | Only modify when adding or removing a dependency that has been explicitly approved. |
| `.env` / `.env.example` | Never write secret values into these files. Never log their contents. Only add new keys when a new environment variable is introduced by the current unit. |
| `src/context/AuthContext.jsx` | Central auth state. Changes here affect every authenticated page. Only modify with explicit instruction. |

## 6. Keeping Documentation in Sync

**Update `architecture.md` when an architectural decision changes.**
If the implementation of a unit reveals that a decision in `architecture.md` needs to change — for example, a new middleware is added, or a storage decision is revised — update `architecture.md` in the same step as the implementation. Do not defer it.

**Update `project-overview.md` when in-scope or out-of-scope items change.**
If a feature is moved from out-of-scope to in-scope (or vice versa), update the relevant section of `project-overview.md` before implementing that feature.

**Update `implementation_plan.md` task checklist as you work.**
Mark tasks as in-progress before starting them and as complete immediately after verifying them. Do not batch-update the checklist at the end of a session.

**Do not let code and documentation contradict each other.**
If the code does one thing and the documentation describes another, fix the documentation in the same commit as the code. A codebase where the docs are wrong is worse than one with no docs.

## 7. Verification Checklist

Complete every item on this checklist before marking a unit as done and moving to the next one. Do not skip items. Do not move forward with partial checks.

### Backend Unit Checklist
- [ ] The route is declared in the correct router file (`routes/`).
- [ ] The controller only parses the request and calls a service. It contains no Prisma queries.
- [ ] The service contains all Prisma queries for this unit. No Prisma calls exist outside the service.
- [ ] A Zod schema validates the request body before the controller runs.
- [ ] `authGuard` runs before `roleGuard` on every protected route. Both are present where required.
- [ ] Ownership is checked before any mutation (update, delete, shortlist, reject).
- [ ] The `password` field is excluded from every Prisma query that returns a User object.
- [ ] The `phone` field is excluded from every response that is not a confirmed shortlisted applicant request.
- [ ] All error paths return the correct HTTP status code: 400 for bad input, 401 for unauthenticated, 403 for unauthorized, 404 for not found, 500 for unexpected errors.
- [ ] The route has been tested manually with at least one success case and one failure case (e.g. missing field, wrong role, wrong owner).

### Frontend Unit Checklist
- [ ] The page or component fetches data only through functions defined in `src/api/`. No raw `axios` calls exist in page files.
- [ ] Loading and error states are handled visibly — the user sees feedback, not a blank screen.
- [ ] The page is not accessible to a user with the wrong role (route guard is in place).
- [ ] The layout renders correctly at 375px width (mobile) and 1280px width (desktop). No horizontal scroll at 375px.
- [ ] No hardcoded strings that belong in environment variables (API URLs, secrets).
- [ ] The component does not store server data in local state if it can be derived from the API response.

### Database / Schema Unit Checklist
- [ ] The change is declared in `prisma/schema.prisma`.
- [ ] `prisma migrate dev` was run and completed without errors.
- [ ] The migration file in `prisma/migrations/` was auto-generated (not hand-written).
- [ ] `prisma generate` was run after the migration to update the Prisma client.
- [ ] Existing seed data or development data is not broken by the migration.

### Before Moving to the Next Unit
- [ ] All checklist items above (for the applicable unit type) are complete.
- [ ] No console errors or warnings are present in the browser or server terminal that are caused by the current unit.
- [ ] The `implementation_plan.md` task for this unit is marked complete.
- [ ] If any file outside the planned scope was modified, that change is documented and explained.
- [ ] If any architectural decision was revised, `architecture.md` has been updated.
